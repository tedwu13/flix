Layouts

Exercises

Objective

Every good web app has some sort of consistent layout across all the pages: a header at the top, a footer at the bottom, and perhaps even a sidebar next to the main content. What's considered fashionable in terms of shapes, sizes, colors, and other design choices tends to change (usually about the time you launch your app). But when it comes to the layout of your app, consistency never goes out of style.

And to keep things consistent, we need to be able to put all the common layout elements in one definitive place. In Rails, that's called the layout file. A layout file is just a regular ERb view template that "wraps" the other view templates in the application. In other words, the layout generates HTML for the header, footer, and sidebar that surrounds the HTML generated by an action's view template. That way, we end up with a complete HTML document to send back to the browser. By having the layout in a single file, we can easily change the layout of our app.

We've gotten this far using the default HTML layout that was generated by Rails. That helped us get an app up and running quickly. Now it's time to start customizing the layout to better suit our app. We'll start by adding a consistent header, footer, and sidebar on all pages. Then we'll decompose the layout into partials for better organization.

Given what you already know about partials, this turns out to be a fairly easy exercise. So let's jump right into it!

Feature Specs

Although we could write a feature spec that verifies each page has a header, footer, and sidebar section, it's not a very high-value test. However, after changing the layout, it's a good idea to run the existing specs just to make sure nothing broke.

1. Review the Default Layout

Let's start by reviewing what we get for free...

  1. Open the default layout file in app/views/layouts/application.html.erb and you should see the following: 

    <!DOCTYPE html>
<html>
  <head>
    <title>Flix</title>
    <%= csrf_meta_tags %>
    <%= stylesheet_link_tag    'application', media: 'all', 'data-turbolinks-track': 'reload' %>
    <%= javascript_include_tag 'application', 'data-turbolinks-track': 'reload' %>
  </head>

  <body>
    <%= yield %>
  </body>
</html>

    Notice that the layout file is a mix of HTML and ERb tags, just like a regular view template.

  2. To see what it generates, browse to any page in the app and view the generated source. What you'll see is the HTML that was generated by a particular view template, index for example, is surrounded by the HTML generated by the layout template. Let's unpack how that happens.

    So far, so good. That's all fairly straight-forward.

    Next comes the most non-intuitive part of the layout, which is also the most important part: 

    <%= yield %>

    The yield returns the output (HTML) that an action generates. That output is then substituted right into the layout file because the <%= %> ERb tag was used. For example, if you visit the index page, the yield line in the layout file is replaced by the HTML that the index.html.erb template generates: 

    <h1>4 Movies</h1>

<article class="movie">
  ...
</article>

...

    So whichever page you visit, the main content changes but the surrounding layout remains consistent. That's the essence of a layout file!

  3. Just as a quick test, go ahead and change the contents of the title to a custom name for your app. Then refresh and you should see the new page title appear at the top of your browser window for every page in the app.

2. Add a Header, Footer, and Sidebar

Now let's start customizing. To keep the marketing folks happy, each page needs some marquee header text at the top. And to keep the legal folks happy (a tall order), each page needs a footer at the bottom with legalese. And just for fun (to keep the rest of us happy), each page also needs a sidebar section with common navigation links.

  1. Start by adding an HTML5 header element above the main content. Put the name of the app in the header. If you want to tart it up with some inline CSS styles, go for it. We've included some basic styling in the answer if you're new to CSS.

    <header style="padding: 5px; color: #F0F0F0; background: #237CBE;">
  <h1>Flix</h1>
</header>

  2. In the browser, make sure the header shows up on every page before moving on.

  3. Next, add an HTML5 footer element below the main content. Put a copyright notice and a link back to your company in the footer. Again, don't worry about the styling. Seriously, who reads the fine print anyway?

    <footer style="text-align: center; padding: 5px; color: #237CBE; background: #F0F0F0">
  <p>
    Copyright &copy; 2016
    <%= link_to 'The Pragmatic Studio','http://pragmaticstudio.com' %>
  </p>
</footer>

3. Decompose Into Partials

As your application grows, the header, sidebar, and footer sections can quickly become fairly complex on their own. And as things start to accumulate, the layout file can morph into an unrecognizable jumble of HTML and ERb tags. To help keep that under control, it's generally a good practice to extract sections of views (including layouts) that logically go together into partials.

So let's do ourselves a favor now by decomposing the layout file into separate header, footer, and sidebar partials. When we're done, you won't see any visible changes on the site. But you'll feel better knowing the internal design is cleaner and easier to maintain.

  1. Create a file named _header.html.erb in the app/views/layouts directory. Copy the header section from the application.html.erb layout file and paste it into the new _header.html.erb partial.

  <h1>Super-Heroic Flix</h1>
</header>

  2. Then, back in the layout file, replace the header section with a call to render the layouts/header partial.

    By default, render looks for the partial in the same directory as the current view template being rendered. If there’s a slash (/) in the partial name, render treats the first part as the directory name and the second part as the file name. Remember that you'll need to use the ERb tag <%= %> (with an equal sign) in order for the rendered partial to get substituted back into the layout file.

    <%= render "layouts/header" %>

  3. Next up is the sidebar. Create a file named _sidebar.html.erb in the app/views/layouts directory. Copy the sidebar section from the application.html.erb layout file and paste it into the new _sidebar.html.erb partial.

     </li>
     <li>
       <%= link_to 'Add New Movie', new_movie_path %>
     </li>
   </ul>
  </nav>
</aside>

  4. Then, back in the layout file, render the layouts/sidebar partial.

Bonus Round

Fundraising App

Someone told the Marketing department about your fundraising app, and they now have the following list of requirements:

  • The header on each page should show the name of your application.

  • The footer on each page should show a copyright notice, your name (so Marketing knows who to submit future requirements to), and a link to the company's website.

  • The sidebar on each page should have two links: "All Projects" and "Add New Project".

Surprisingly, Marketing didn't send along any requirements for fonts, colors, or other styling concerns, so those decisions are entirely up to you. Put on your creative hat and go for it! (But make sure all your specs pass when you're done!)

Wrap Up

And that's all there really is to layouts! They're just regular ERb templates that "wrap" the other view templates so your application is displayed in a consistent layout.

Structuring the app around a basic layout is a good start. But once you've been introduced to the layout file, you start thinking about colors, sizes, fonts, and so on. Yup, this is the point where application development crosses over into web design. That's a huge topic that's beyond the scope of this course, but in the next section we'll have a quick look at styling the app with CSS.

Dive Deeper

The application.html.erb layout file is the application-wide layout. By default, it's applied to every action in the application. But Rails offers lots of options for rendering a custom layout file depending on the controller, action, or even arbitrary application logic. To learn more about custom layouts, refer to 2.2.12 Finding Layouts in the Rails Guide: Layouts and Rendering.
